# CyberAgent Architecture and Agent Flow

This document explains how the current CyberAgent project works, how context is passed between agents, and what each part of the system is doing in the codebase today.

## 1. How Is Our Classification Agent Identifying the Type of Attack?

The Classification Agent is implemented in the `threat_classification()` function in [backend/agents.py](/Users/rohithborana/stuf/MajProj/cyber-agents/backend/agents.py).

It does not use a trained machine learning model. Instead, it identifies the attack type by reading the anomaly evidence produced by the Anomaly Detection Agent.

It looks at:
- the number of distinct source IPs
- the number of recent access log events
- the number of failed authentication attempts
- the number of destination ports touched
- the total packets observed

Based on those signals, it maps the anomaly to one of three attack types:

- `DDoS`
  Chosen when there are many source IPs, a large request burst, and a high packet total.

- `BruteForce`
  Chosen when there are repeated failed authentication attempts from the same source.

- `PortScan`
  Chosen when one source touches many ports in a short interval.

So the current attack type is determined by rule-based analysis of telemetry, not by a model artifact.

## 2. How Is It Determining the Confidence Score?

The confidence score is also computed inside `threat_classification()` in [backend/agents.py](/Users/rohithborana/stuf/MajProj/cyber-agents/backend/agents.py).

The system assigns a confidence range based on the detected anomaly type:

- `DDoS`: random value in a high range
- `BruteForce`: random value in a high range
- `PortScan`: random value in a high range
- fallback suspicious activity: lower range

After choosing the predicted class, it creates a confidence distribution across all three classes. The predicted class gets the highest score, and the others get lower values.

Important note:

This confidence is currently part of the simulation logic. It is meant to represent how confident the blue-team pipeline is in the classification result, but it is not coming from a real trained classifier.

## 3. What Is This Telemetry and Why Are We Using It?

In this project, telemetry means the raw technical signals that the blue-team agents monitor and analyze.

Our telemetry consists of:
- access logs
- auth logs
- network logs

Telemetry is used because real cybersecurity systems do not start with a clean incident object. They start with noisy raw evidence such as requests, failed logins, connections, and packet flow summaries.

We use telemetry in CyberAgent because it makes the project more realistic:

- Red Team Agent generates raw log entries.
- Log Monitor Agent reads those logs.
- Anomaly Detection Agent extracts suspicious patterns from them.
- Classification Agent turns those patterns into a structured incident.

That means telemetry is the bridge between simulated attack activity and blue-team decision making.

## 4. What Are the Mitigation Commands and What Does Each One Do?

The mitigation commands are generated by the Response Planning Agent in `threat_resolve()` in [backend/agents.py](/Users/rohithborana/stuf/MajProj/cyber-agents/backend/agents.py).

If Gemini fails, the system falls back to a hardcoded plan with three commands.

### Command 1
`iptables -A INPUT -s <primary_ip> -j DROP`

What it does:
- adds a firewall rule to the Linux `INPUT` chain
- matches packets coming from the suspicious source IP
- drops them immediately

Why it helps:
- prevents further packets from that source from reaching the host

### Command 2
`ufw limit <target_port>/tcp`

What it does:
- applies a rate limit rule for the targeted TCP port using UFW
- slows repeated or abusive connection attempts

Why it helps:
- reduces the pressure on the exposed service
- is useful when the target service is being hit too aggressively

### Command 3
`fail2ban-client set sshd banip <primary_ip>`

What it does:
- tells Fail2ban to ban the suspicious source IP for the `sshd` jail

Why it helps:
- blocks repeated abusive authentication traffic through the intrusion-prevention layer

Important note:

These are real Linux commands, but in this project they are not executed on the host. They are generated as the recommended plan, and the Action Agent simulates their execution.

## 5. What Happens During Action and How Does It Actually Mitigate?

The Action Agent is implemented in `action()` in [backend/agents.py](/Users/rohithborana/stuf/MajProj/cyber-agents/backend/agents.py).

It reads:
- the classified attack context
- the mitigation plan
- the approval decision

### If approved
The Action Agent:
- iterates through the mitigation steps
- marks each step as `EXECUTED`
- assigns a fake `execution_time_ms`
- records the blocked IPs
- sets the final status to `MITIGATED`

### If rejected
The Action Agent:
- does not execute the steps
- creates a manual intervention ticket
- assigns the case to `security-team@company.com`
- sets the status to `MANUAL_INTERVENTION_REQUIRED`

How mitigation works in the current project:

- It is simulated mitigation, not live infrastructure control.
- The system records what would be executed and shows the result in the UI.
- This is useful for demonstrating workflow, orchestration, and response planning without actually changing firewall rules on the machine.

## 6. What Are Access, Auth, and Network Logs? What Is the Difference?

These logs are generated by the Red Team Agent in [backend/red_team.py](/Users/rohithborana/stuf/MajProj/cyber-agents/backend/red_team.py), and then monitored by the Log Monitor Agent.

### Access Logs
These represent website or web server requests.

Examples of information they contain:
- source IP
- HTTP method
- request path
- response status
- response size
- user agent

Why they matter:
- useful for spotting request floods, suspicious scanning of URLs, or repeated hits to sensitive endpoints

### Auth Logs
These represent authentication activity.

Examples of information they contain:
- service name
- source IP
- username
- result such as `FAILED`
- target port

Why they matter:
- useful for detecting brute-force login attempts and repeated authentication failures

### Network Logs
These represent lower-level connection or packet-flow summaries.

Examples of information they contain:
- source IP
- destination IP and port
- protocol
- packet count
- byte count
- flags such as `SYN`

Why they matter:
- useful for spotting port scanning, traffic spikes, and unusual connection behavior

### Difference Between Them

- access logs show web request behavior
- auth logs show login/authentication behavior
- network logs show connection and flow behavior

Together, they give the blue-team agents multiple views of the same attack.

## 7. Why Do We Have Primary IP and Target IP?

The classified attack object contains both a primary source IP and a target IP.

### Primary IP
This is the main suspicious source IP selected from the observed telemetry.

Why we use it:
- to identify the most important attacking source
- to generate mitigation commands against a specific IP
- to summarize the incident clearly in reports

### Target IP
This is the destination host being attacked.

Why we use it:
- to know which server or service is under pressure
- to identify the affected target port
- to provide context for planning and reporting

In short:
- primary IP = attacker side
- target IP = victim side

## 8. How Is the Orchestration Happening?

The orchestration layer is implemented with LangGraph in [backend/agents.py](/Users/rohithborana/stuf/MajProj/cyber-agents/backend/agents.py).

The project defines a shared `AgentState` object and passes it through a sequence of graph nodes.

### Detection Graph
`log_monitor -> anomaly_detection -> threat_classification -> threat_resolve -> END`

### Action Graph
`action -> report -> END`

Each node:
- reads values already stored in the shared state
- adds new outputs to the state
- passes the updated state to the next node

The FastAPI backend in [backend/main.py](/Users/rohithborana/stuf/MajProj/cyber-agents/backend/main.py) starts these graphs using `run_in_executor()` because LangGraph invocation is synchronous.

This means the orchestration is:
- explicit
- state-driven
- observable in the frontend through WebSocket events

## 10. Full End-to-End Flow

Here is the full project flow as implemented today.

1. A simulation starts from the FastAPI backend in [backend/main.py](/Users/rohithborana/stuf/MajProj/cyber-agents/backend/main.py).
2. The Red Team Agent writes synthetic raw log entries into runtime log files.
3. The initial LangGraph state is created with the red-team simulation context.
4. The Log Monitor Agent reads the latest access, auth, and network logs.
5. The Anomaly Detection Agent analyzes those logs and flags suspicious behavior.
6. The Classification Agent converts that evidence into a structured incident object.
7. The Response Planning Agent uses that incident context to generate mitigation steps with Gemini.
8. The system pauses and waits for human approval in the dashboard.
9. If approved, the Action Agent simulates mitigation execution.
10. If rejected, the Action Agent creates a manual intervention path.
11. The Reporting Agent generates a final incident report with Gemini.
12. The backend broadcasts each stage over WebSocket so the frontend can show the full progression.

## 11. Where Is Context Passed Between Agents?

Context is passed through the shared `AgentState` object in [backend/agents.py](/Users/rohithborana/stuf/MajProj/cyber-agents/backend/agents.py).

The main state fields are:
- `simulation`
- `telemetry`
- `anomaly`
- `classification`
- `mitigation_plan`
- `approval_status`
- `action_result`
- `incident_report`

The flow of context is:

1. Red Team Agent creates `simulation`
2. Log Monitor Agent reads `simulation` and adds `telemetry`
3. Anomaly Detection Agent reads `telemetry` and adds `anomaly`
4. Classification Agent reads `anomaly` and `telemetry` and adds `classification`
5. Response Planning Agent reads `classification` and adds `mitigation_plan`
6. Action Agent reads `classification`, `mitigation_plan`, and `approval_status`
7. Reporting Agent reads `classification` and `action_result`

This shared-state flow is the core of the multi-agent orchestration.

## 12. Final Notes

What is real in this project today:
- distinct agents with distinct roles
- real shared context passed through LangGraph
- real raw log generation and monitoring flow
- human approval gate
- WebSocket-based stage updates
- Gemini used only for planning and reporting

What is still simulated:
- the raw logs are synthetic rather than collected from a production server
- attack classification logic is rule-based rather than model-based
- mitigation commands are recommended and simulated, not executed on live infrastructure

That makes CyberAgent a realistic multi-agent workflow prototype rather than a production SOC platform.
